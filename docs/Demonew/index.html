Component library

Each component uses token-driven anatomy, variants, states, sizes, and accessibility behavior.

Buttons

Anatomy: container, label, optional leading icon, optional trailing icon. Sizes use fixed heights with token padding.

  • Default height: 40px. Compact: 32px. Large: 48px.
  • Focus must use visible ring, not color shift only.
  • Ghost button is for low-emphasis actions on neutral surfaces.

Inputs, select, textarea

  • Anatomy: label, control, helper text, error text.
  • Use labels above inputs; never rely on placeholder as label.
  • Error states use danger border plus supportive message text.

Data components

Data-heavy interfaces prioritize scanability, alignment, and minimal visual noise.

Data table rules

  • Row height: 48px default, 40px dense.
  • Header style: muted surface, medium weight, bottom border.
  • No zebra rows. Hover row highlight only.
  • IDs, names, and labels left-align. Currency, counts, and percentages right-align.

Pagination

  • Use for 7+ pages or server-driven datasets.
  • Current page uses primary fill and white text.

Sorting indicators

  • Place sorting icons to the right of header text.
  • Only one active sort per table unless explicitly multi-sort.

UI examples

Wireframe-level compositions using the defined tokens and component rules.

SaaS dashboard

Uses page header, KPI cards, one primary chart, one supporting data table, and sidebar navigation. Apply 24px grid gap and 48px section separation.

Analytics

Uses filters row, chart cards, segmented tabs, and a detailed table. Keep numeric alignment strict and charts no more than two per row.

User profile

Uses profile summary card, editable form sections, status badges, and an activity table. Keep forms in single-column layout unless on wide desktop.

Settings

Uses sectioned cards with labels and helper text, toggle-like actions replaced by buttons where needed, and warning alert before sensitive changes.

Billing

Uses plan summary card, invoice table, payment method card, usage alert, and modal for destructive changes. Put billing totals in right-aligned value columns.

AI rules

Strict generation rules for any AI producing UI with this system.

  • Must use only defined tokens for color, spacing, typography, radius, and shadows.
  • Must prefer cards for summaries and forms, tables for dense multi-row data, alerts for stateful notices, and modals only for blocking confirmations.
  • Must keep a single dominant action per surface region.
  • Must use accessible labels, semantic landmarks, visible focus states, and contrast-safe text/background pairings.
  • Never introduce decorative gradients, extra accent colors, oversized shadows, or decorative icons that do not aid scanning.
  • Never mix unrelated densities in one region. Stay compact or comfortable within the same surface.
  • Never create nested cards beyond one level or place more than four KPI cards in a single row.
